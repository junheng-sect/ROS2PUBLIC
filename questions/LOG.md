## 2026-03-07 22:34:30 CST

- 问题：`PX4` 中的 `Return` 模式实现原理是什么？
- 解答：`Return` 本质上就是 `RTL(Return-to-Launch)` 任务状态机。触发后，飞控会先判断当前位置、返航点、当前高度和是否允许直接返航；然后通常按“必要时先爬升到安全返航高度 -> 水平飞回返航点上方 -> 下降到指定高度 -> 等待/降落/解锁”这一序列执行。多旋翼一般会先升到 `RTL_RETURN_ALT`（若当前更高则保持当前高），飞到 Home 或 Rally Point，再按 `RTL_DESCEND_ALT` 下降，最后依据参数执行悬停、着陆或解锁关机。固定翼/VTOL 会结合航线半径、进近方式和机型逻辑处理，不完全等同于多旋翼直线返航。返航触发源可以是模式切换、失联、低电量、任务异常等，本质上都是把导航控制权切换到 PX4 内部的自动返航状态机。官方文档：`https://docs.px4.io/main/en/flight_modes_mc/return`、`https://docs.px4.io/main/en/config/safety.html#return-mode-settings`。

## 2026-03-07 22:36:27 CST

- 问题：返航目标点是怎么获得的？
- 解答：PX4 的返航目标点默认来自 `Home Position`。`Home` 一般在飞行器满足定位条件并解锁/起飞前后由飞控记录，通常就是起飞点附近的全局位置与高度参考。进入 `Return/RTL` 后，导航模块会优先根据配置判断是否使用 `Rally Point` 或其他可用安全返航点；如果没有这类替代点，目标就是 `Home Position`。所以本质上是“先看是否有更合适的返航点（如 Rally），否则回 Home”。若 `Home` 本身无效或定位源不可用，RTL 能力也会受限。参考：`https://docs.px4.io/main/en/advanced_config/home_position.html`、`https://docs.px4.io/main/en/flight_modes_mc/return`。

## 2026-03-07 22:39:32 CST

- 问题：`Home Position` 在 `PX4` 里具体是在哪个模块设置的、什么时候更新、`ROS 2/MAVROS` 怎么读到它？
- 解答：在 PX4 内部，`Home Position` 由 `commander` 模块负责维护，核心逻辑在 `HomePosition` 类中完成；它会订阅定位估计和起飞/解锁状态，在满足条件时设置或更新 home。更新时机通常是“定位有效且允许设置 home 时”，常见是在解锁前后、起飞前后首次获得可靠全局位置时；若参数允许空中设置（如 `COM_HOME_IN_AIR`），飞行中也可能更新。对外发布时，PX4 通过 MAVLink `HOME_POSITION` 消息发送，MAVROS 将其转换成 `mavros_msgs/msg/HomePosition`；ROS 2 侧通常直接订阅 `/mavros/home_position/home` 读取。若要主动设置 home，可通过 MAVROS 服务 `mavros_msgs/srv/CommandHome`（常见服务名 `/mavros/cmd/set_home`）发送当前位置或指定经纬高。参考：`https://docs.px4.io/main/en/advanced_config/home_position.html`、`https://docs.px4.io/main/en/advanced_config/parameter_reference.html`（`COM_HOME_IN_AIR`）、`https://mavlink.io/en/messages/common.html#HOME_POSITION`。

## 2026-03-07 22:44:37 CST

- 问题：继续讲 `Home Position` 和 `vehicle_global_position`、`local_position_ned`、`EKF` 原点这几个概念的区别。
- 解答：这几个概念分属“返航参考点”“全球位置估计”“局部坐标估计”“局部坐标原点”四层。`Home Position` 是任务/安全逻辑使用的返航参考点，本质是一个被保存的特殊位置；`vehicle_global_position` 是 EKF 融合后的实时全球位置解算结果，内容是当前飞行器经纬高，不等于 home；`local_position_ned`（对应 `VehicleLocalPosition`）是当前飞行器在局部 NED 坐标系下的位置/速度估计，其中 `x/y/z` 是相对“局部原点”的实时坐标；`EKF` 原点就是这个局部坐标系的参考零点，通常在 EKF 启动并建立局部参考系时确定。关键区别是：`Home Position` 主要给 RTL 等导航任务使用，`vehicle_global_position` 描述“我现在在地球上的哪”，`local_position_ned` 描述“我相对局部原点在哪”，`EKF` 原点则定义了这个局部坐标系本身。`Home` 和 `EKF` 原点可能接近，但不是同一个概念，也不要求始终重合。参考：`https://docs.px4.io/main/en/advanced_config/home_position.html`、`https://docs.px4.io/main/en/msg_docs/VehicleGlobalPosition.html`、`https://docs.px4.io/main/en/msg_docs/VehicleLocalPosition.html`、`https://docs.px4.io/main/en/advanced_config/tuning_the_ecl_ekf.html`。

## 2026-03-07 22:48:08 CST

- 问题：`home` 点刚需 `GPS` 吗？我如果要做一个返航 `home` 点的程序，是否刚需 `GPS`？
- 解答：不一定。若说“PX4 原生 `Return/RTL` 模式”，官方要求的是有效的 `global 3D position estimate`，这通常来自 GPS，但也可能由其他定位源配合推断得到，所以不是机械意义上的“必须插 GPS 模块”；不过在大多数真实室外场景里，GPS 是最常见也最直接的来源。若你是“自己写一个返航到 home 点的程序”，那就更不刚需 GPS，只要你有稳定、连续、可闭环的位姿参考，并且 home 点和当前位姿都在同一坐标系下，就可以做返航，例如基于 VIO、UWB、MoCap、SLAM、本地里程计都可以。真正的刚需不是 GPS，而是“可靠的位置参考 + 已定义的 home 点 + 可执行的导航控制”。如果没有 GPS，又想直接用 PX4 自带 RTL，就要看你是否给 PX4 提供了可被其接受的全局位置能力；如果只是自己写上层返航控制逻辑，则完全可以只依赖局部坐标系实现“回 home”。参考：`https://docs.px4.io/main/en/flight_modes_mc/return`、`https://docs.px4.io/main/en/advanced_config/home_position.html`。

## 2026-03-07 23:57:02 CST

- 问题：`PX4` 的落地检测机制是什么样的？
- 解答：PX4 的落地检测不是单一开关，而是一个分阶段判定机制，常见状态包括 `ground_contact`、`maybe_landed`、`landed`。对多旋翼而言，系统会综合推力/油门是否很低、竖直速度是否很小、水平运动是否很小、角速度是否足够低、是否仍在受控下降等条件进行判断。通常先进入“接地接触” (`ground_contact`)，再进入“可能已落地” (`maybe_landed`)，最后在条件持续满足后判定为“已落地” (`landed`)。这个结果会被 `commander`、自动降落、自动 disarm、failsafe 等逻辑使用，因此它本质上是飞控安全状态机的一部分，而不是简单看高度是否为零。不同机型（多旋翼、固定翼、VTOL）阈值和触发条件会不同，多旋翼最依赖低推力、低速度和低角速度联合判断。参考：`https://docs.px4.io/main/en/advanced_config/land_detector.html`、`https://docs.px4.io/main/en/msg_docs/VehicleLandDetected.html`。

## 2026-03-08 00:47:40 CST

- 问题：状态机是什么？
- 解答：状态机可以理解为“把程序运行过程拆成若干明确状态，并规定每种状态在什么条件下切换到下一种状态”的控制方法。它的核心是三件事：当前处于什么状态、每个状态下该做什么、满足什么条件后切换状态。比如无人机自动降落可以分成“搜索目标、对准目标、下降、接地、落地完成”几个状态，每个状态执行不同控制逻辑，条件满足后再切换。这样做的好处是逻辑清晰、便于调试，也更适合处理复杂流程。

- 问题：AI 提示词 `prompt` 是什么？
- 解答：`prompt` 就是你给 AI 的输入指令，可以理解为“你希望 AI 做什么、按什么要求做、输出成什么样”的说明。它不只是一个问题，也可以包含背景、角色、限制条件、格式要求、示例等。`prompt` 写得越清楚，AI 输出通常越稳定、越接近你的目标。例如“帮我写论文摘要”和“请用本科论文风格、200字左右、围绕无人机视觉降落系统写一个摘要”就是不同质量的 prompt，后者约束更明确，结果通常更可用。

- 问题：详细介绍一下 `Linear MCP` 和 `skill`。
- 解答：`Linear MCP` 和 `skill` 都是在给 AI “扩能力”，但层级不同。`MCP`（Model Context Protocol）是一个标准协议，用来让 AI 连接外部工具和数据源；`Linear MCP` 就是基于这个协议，把 Linear 的 issue、project、comment 等能力暴露给 AI，所以 AI 可以读写 Linear 中的数据。它更像“外部系统接口层”。而 `skill` 更像“任务工作流封装层”，用于把某类任务的说明、步骤、模板、约束整理成一个可复用能力包，告诉 AI 遇到某类任务时该怎么做。前者重点是“让 AI 接到外部系统”，后者重点是“让 AI 按既定方法完成任务”。简单说：`Linear MCP` 解决“AI 能访问什么工具”，`skill` 解决“AI 拿到工具后该怎么做事”。参考：MCP 官方文档 `https://modelcontextprotocol.io/`，Linear 官方 MCP 说明 `https://linear.app/docs/mcp`。

## 2026-03-08 18:20:19 CST

- 问题：`Windows` 系统能访问 `Ubuntu` 的文件吗？
- 解答：能，但取决于你是哪种使用场景。如果是 `WSL`，Windows 可以直接通过资源管理器访问 Ubuntu 文件，例如 `\\\\wsl$\\发行版名\\`；如果是双系统安装的 Ubuntu，Windows 一般不能原生稳定读写 Linux 的 `ext4` 分区，通常需要第三方工具或通过 Ubuntu 共享出来；如果是两台机器，也可以通过局域网共享、SSH、SFTP 等方式访问。所以结论不是绝对“能”或“不能”，而是要看你用的是 WSL、双系统，还是网络共享方式。

- 问题：我是双系统。
- 解答：如果你是 `Windows + Ubuntu` 双系统，那么 Windows 对 Ubuntu 分区的访问通常受限。原因是 Ubuntu 常用的文件系统是 `ext4`，Windows 默认不能像读 `NTFS` 一样稳定原生读写它。因此一般结论是：Windows 不能方便、稳定地直接访问 Ubuntu 系统盘里的文件。可行办法通常有三类：一是在 Ubuntu 里把文件放到一个 Windows 和 Ubuntu 都能访问的数据分区；二是用 Ubuntu 开启共享，通过局域网、SFTP、SSH 让 Windows 访问；三是借助第三方 `ext4` 读写工具，但不建议长期把重要开发文件放在这种跨系统直接写入场景里，容易有兼容和损坏风险。

## 2026-03-09 13:13:45 CST

- 问题：`/mavros/setpoint_attitude/cmd_vel` 和 `/mavros/setpoint_velocity/cmd_vel` 这两个话题有什么不同？
- 解答：这两个话题控制层级不同。`/mavros/setpoint_velocity/cmd_vel` 是速度控制入口，通常发布的是机体系或局部坐标系下的线速度/角速度期望，飞控会继续在内部做速度环、姿态环和混控，更偏“高层控制”。而 `/mavros/setpoint_attitude/cmd_vel` 属于姿态控制相关入口，语义更接近“角速度指令”或与姿态控制器直接相关的输入，控制层级更低，更接近姿态内环。简单说：前者是在告诉飞控“我要往哪边以多快速度飞”，后者更像是在告诉飞控“机体姿态或角速度该怎么变化”。做位置/视觉闭环跟踪时，通常优先用 `setpoint_velocity`；只有在你明确要自己接管更低层姿态行为时，才会考虑 `setpoint_attitude` 相关话题。

## 2026-03-09 13:19:42 CST

- 问题：`setpoint_velocity/cmd_vel` 的输入量有哪些？是在哪个坐标系下的？
- 解答：`/mavros/setpoint_velocity/cmd_vel` 常用消息类型是 `geometry_msgs/msg/TwistStamped`，核心输入量是 `twist.linear.x/y/z` 和 `twist.angular.z`。其中 `linear.x/y/z` 表示三轴线速度期望，`angular.z` 通常表示偏航角速度期望；`angular.x/y` 一般不作为这个接口的主要控制量。坐标系由 `setpoint_velocity` 插件的 `mav_frame` 参数决定，在当前 MAVROS 默认配置里通常是 `LOCAL_NED`。但从 ROS 使用角度看，MAVROS 会在 ROS 常用坐标系和 MAVLink 坐标系之间做转换，所以用户经常按 ROS 侧的 `ENU` 语义来理解本地坐标速度命令。若把 `mav_frame` 改成机体系，如 `BODY_NED/BODY_OFFSET_NED`，则速度会变成机体系前右下语义。默认情况下更适合把它理解为“局部坐标系下的三轴速度 + 偏航角速度”接口。

## 2026-03-09 16:28:52 CST

- 问题：GPS 的数据时间长了会漂吗？尤其是高度方向。
- 解答：会。GPS 不是“固定住就永远不变”的绝对真值，静止时位置和高度都会随时间抖动或缓慢漂移，其中高度方向通常比水平位置更差、更容易漂。原因主要是卫星几何关系、时钟误差、多路径反射、大气误差和接收机噪声共同作用。工程上常见现象是：水平位置会在一个范围内来回跳动，而高度更容易出现更大的波动或慢漂，所以单纯依赖 GPS 高度做精确悬停或精确降落通常不够。对 PX4 实机来说，z 方向一般更依赖气压计、测距仪或视觉/激光等辅助，高精度场景则用 RTK、视觉定位或融合更多传感器。若你拿普通 GPS 直接长期看高度，出现明显波动是正常现象。

- 问题：`Position` 模式下，摇杆的输入映射的是目标速度吗？
- 解答：基本上是。对 PX4 多旋翼 `Position` 模式来说，横滚/俯仰摇杆通常不会直接绕过位置控制器去控姿态，而是先转成水平速度目标；偏航摇杆对应偏航角速度或航向变化，油门杆通常对应垂向速度或高度变化请求。飞控再通过位置环和速度环把这些目标转成姿态与推力。中位回杆后，水平速度目标回到零，飞控会把当前位置“接住”，继续做定点悬停。所以更准确地说：`Position` 模式下摇杆主要映射的是“期望运动量（以速度为主）”，不是直接映射成最终姿态输出。
